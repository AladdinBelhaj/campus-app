# Lab09 — Exposer des applications sur OpenShift (Routes, NodePort, LoadBalancer)

---

## Contexte

Dans ce lab, vous travaillez sur une plateforme composée de trois types d'applications aux besoins d'exposition différents :

- Une **application web** Python qui répond en HTTP — le cas le plus courant
- Une **base de données PostgreSQL** qui parle TCP pur, sans HTTP — les Routes OpenShift ne peuvent pas l'exposer
- Une **application NGINX avec TLS natif** — le pod gère lui-même le chiffrement, sans déléguer au routeur

Chaque cas nécessite une stratégie d'exposition différente. Ce lab vous permet de comprendre pourquoi et comment choisir la bonne approche.

---

## Objectifs

- Créer et configurer des **Routes HTTP et HTTPS**
- Comprendre les modes **Edge, Passthrough et Re-encrypt**
- Exposer des services **non-HTTP** via NodePort et LoadBalancer
- Utiliser les **annotations HAProxy** pour contrôler le comportement du routeur
- Vérifier l'accès externe depuis `curl`

## Étape 0 — Préparer le namespace

Avant de commencer, il faut se positionner dans le bon namespace et identifier le **domaine wildcard** du cluster. Ce domaine est utilisé par OpenShift pour générer automatiquement les hostnames des Routes. Toutes vos routes auront la forme `<nom>.<domaine-wildcard>`, par exemple `webapp.apps.rosa-cluster.example.com`.

```bash
oc project expose

oc get ingresses.config.openshift.io cluster -o jsonpath='{.spec.domain}' && echo
```

Résultat attendu :
```
apps.rosa-cluster.example.com
```

Notez ce domaine, il sera utilisé dans les étapes suivantes.

## Étape 1 — Exposer une application HTTP simple via une Route

### Concept

Une **Route** OpenShift est l'équivalent d'un Ingress Kubernetes, mais avec plus de fonctionnalités (TLS, annotations HAProxy, etc.). Elle est gérée par le routeur HAProxy intégré au cluster. Quand vous créez une Route sur un Service, OpenShift configure automatiquement HAProxy pour rediriger le trafic entrant vers les pods correspondants.

Pour qu'une Route fonctionne, il faut trois éléments :
1. Un **Deployment** qui crée les pods
2. Un **Service** qui regroupe ces pods (le routeur parle au Service, pas aux pods directement)
3. Une **Route** qui expose le Service vers l'extérieur

### 1.1 — Déployer l'application et son Service

L'application est un simple serveur HTTP Python. Le Service de type `ClusterIP` (par défaut) regroupe les pods et leur assigne une IP interne stable. Le routeur OpenShift utilisera ce Service comme backend.

```bash
cat > webapp.yaml << 'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
        - name: webapp
          image: registry.access.redhat.com/ubi9/python-311:latest
          command:
            - /bin/sh
            - -c
            - python3 -m http.server 8080
          ports:
            - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: webapp
spec:
  selector:
    app: webapp
  ports:
    - port: 8080
      targetPort: 8080
EOF

oc apply -f webapp.yaml

# Attendre que les pods soient Running avant de continuer
oc get pods -l app=webapp -w
```

### 1.2 — Créer la Route HTTP

La commande `oc expose service` est le raccourci le plus simple. Elle lit la configuration du Service (port exposé, sélecteur) et crée automatiquement une Route avec un hostname généré à partir du nom du service et du domaine wildcard du cluster.

```bash
oc expose service webapp

# Vérifier le hostname généré automatiquement
oc get route webapp -o jsonpath='{.spec.host}' && echo
```

### 1.3 — Vérifier l'accès

```bash
ROUTE=$(oc get route webapp -o jsonpath='{.spec.host}')

# Le code 200 confirme que la route fonctionne et que le pod répond
curl -o /dev/null -s -w "%{http_code}\n" "http://$ROUTE"
```

Résultat attendu : `200`

## Étape 2 — Route HTTPS avec terminaison Edge

### Concept

La terminaison **Edge** est le mode TLS le plus courant. Le routeur HAProxy reçoit le trafic HTTPS, le **déchiffre**, puis le transmet au pod en HTTP clair. Cela permet au pod de ne pas gérer de certificats — le chiffrement est entièrement pris en charge par le routeur.

```
Client ──HTTPS──► Routeur OpenShift (déchiffre) ──HTTP──► Pod
```

Vous fournissez votre propre certificat TLS au routeur. En production, ce serait un certificat signé par une CA reconnue (Let's Encrypt, DigiCert...). Ici, nous utilisons un certificat auto-signé.

### 2.1 — Générer un certificat TLS auto-signé

Le certificat doit correspondre exactement au hostname de la route. Nous utilisons un fichier de configuration OpenSSL pour inclure un **Subject Alternative Name (SAN)**, requis par les navigateurs modernes.

```bash
# Récupérer le domaine wildcard et construire le hostname cible
DOMAIN=$(oc get ingresses.config.openshift.io cluster -o jsonpath='{.spec.domain}')
HOSTNAME="webapp-edge.${DOMAIN}"
echo "Hostname cible : $HOSTNAME"

# Générer la configuration OpenSSL avec le SAN
cat > openssl-san.cnf << EOF
[ req ]
default_bits       = 2048
prompt             = no
default_md         = sha256
distinguished_name = dn
x509_extensions    = v3_req

[ dn ]
CN = ${HOSTNAME}
O  = Workshop

[ v3_req ]
subjectAltName = @alt_names

[ alt_names ]
DNS.1 = ${HOSTNAME}
EOF

# Générer la clé privée et le certificat auto-signé (valide 1 an)
openssl req -x509 -nodes -days 365 \
  -newkey rsa:2048 \
  -keyout tls.key \
  -out tls.crt \
  -config openssl-san.cnf

# Vérifier que le certificat contient le bon CN et SAN
openssl x509 -in tls.crt -text -noout | grep -E "Subject:|DNS:"

# Stocker le certificat dans un Secret OpenShift
# Le routeur lira ce Secret pour présenter le certificat aux clients
oc create secret tls webapp-tls --cert=tls.crt --key=tls.key
```

### 2.2 — Créer la Route Edge

L'option `--insecure-policy=Redirect` force les clients HTTP à basculer automatiquement en HTTPS (réponse 301). Sans cette option, les deux protocoles seraient accessibles simultanément.

```bash
oc create route edge webapp-edge \
  --service=webapp \
  --cert=tls.crt \
  --key=tls.key \
  --insecure-policy=Redirect

oc get route webapp-edge
```

### 2.3 — Vérifier l'accès HTTPS

```bash
ROUTE=$(oc get route webapp-edge -o jsonpath='{.spec.host}')

# -k ignore l'avertissement du certificat auto-signé
curl -k -o /dev/null -s -w "%{http_code}\n" "https://$ROUTE"

# Vérifier que HTTP redirige bien vers HTTPS (attendu : 301)
curl -o /dev/null -s -w "%{http_code}\n" "http://$ROUTE"

# Inspecter le certificat présenté par le routeur
echo | openssl s_client -connect "${ROUTE}:443" 2>/dev/null | openssl x509 -noout -subject -dates
```

Résultats attendus : `200` pour HTTPS, `301` pour HTTP.

## Étape 3 — Route HTTPS avec terminaison Passthrough

### Concept

En mode **Passthrough**, le routeur ne déchiffre pas le trafic. Il lit uniquement le **nom de domaine dans le header TLS (SNI)** pour router vers le bon service, puis transmet le flux chiffré tel quel au pod. C'est le pod lui-même qui gère le certificat et le déchiffrement.

```
Client ──HTTPS──► Routeur OpenShift (ne déchiffre pas) ──HTTPS──► Pod
```

Ce mode est utile pour les applications qui imposent leur propre TLS : services gRPC avec certificats applicatifs, ou toute app où vous ne voulez pas que le trafic circule en clair entre le routeur et le pod.

La contrainte principale : le pod doit porter lui-même le certificat, le routeur ne peut pas l'injecter.

### 3.1 — Déployer NGINX avec TLS natif

NGINX va gérer le TLS directement. Nous lui montons le certificat via un Secret (le même généré à l'étape 2.1) et la configuration via une ConfigMap.

> **Pourquoi `nginx-unprivileged` ?**
> L'image `nginx:alpine` standard tente au démarrage de créer des répertoires dans `/var/cache/nginx/` et d'écrire dans `/etc/nginx/`. OpenShift bloque ces opérations pour les conteneurs root via les SCC (Security Context Constraints). L'image `nginxinc/nginx-unprivileged` est conçue pour tourner sans root : tous ses répertoires temporaires sont pré-configurés pour un utilisateur non-privilégié.

```bash
cat > nginx-tls.yaml << 'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    pid /tmp/nginx.pid;
    events {}
    http {
      client_body_temp_path /tmp/client_temp;
      proxy_temp_path       /tmp/proxy_temp;
      fastcgi_temp_path     /tmp/fastcgi_temp;
      uwsgi_temp_path       /tmp/uwsgi_temp;
      scgi_temp_path        /tmp/scgi_temp;
      server {
        listen 8443 ssl;
        ssl_certificate     /etc/nginx/ssl/tls.crt;
        ssl_certificate_key /etc/nginx/ssl/tls.key;
        location / {
          default_type text/plain;
          return 200 "Hello from NGINX with native TLS!\n";
        }
      }
    }
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-tls
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx-tls
  template:
    metadata:
      labels:
        app: nginx-tls
    spec:
      securityContext:
        runAsNonRoot: true
      containers:
        - name: nginx
          image: nginxinc/nginx-unprivileged:alpine
          ports:
            - containerPort: 8443
          securityContext:
            allowPrivilegeEscalation: false
            capabilities:
              drop: ["ALL"]
          volumeMounts:
            - name: tls
              mountPath: /etc/nginx/ssl
              readOnly: true
            - name: config
              mountPath: /etc/nginx/nginx.conf
              subPath: nginx.conf
      volumes:
        - name: tls
          secret:
            secretName: webapp-tls
        - name: config
          configMap:
            name: nginx-config
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-tls
spec:
  selector:
    app: nginx-tls
  ports:
    - port: 8443
      targetPort: 8443
EOF

oc apply -f nginx-tls.yaml

oc get pods -l app=nginx-tls -w
```

### 3.2 — Créer la Route Passthrough

En mode Passthrough, aucun certificat n'est fourni à la Route — le routeur ne participe pas au TLS et transmet le flux tel quel.

```bash
oc create route passthrough nginx-passthrough \
  --service=nginx-tls \
  --port=8443

oc get route nginx-passthrough
```

### 3.3 — Vérifier l'accès

```bash
ROUTE=$(oc get route nginx-passthrough -o jsonpath='{.spec.host}')

# Le trafic arrive directement au pod, qui répond avec son propre certificat
curl -k "https://$ROUTE"

# Confirmer que le certificat vient du pod (même CN que celui généré à l'étape 2.1)
echo | openssl s_client -connect "${ROUTE}:443" 2>/dev/null | openssl x509 -noout -subject
```

Résultat attendu : `Hello from NGINX with native TLS!`

## Étape 4 — Annotations HAProxy sur une Route

### Concept

Le routeur OpenShift est basé sur **HAProxy**. Les Routes acceptent des annotations qui permettent de configurer finement son comportement : délais, algorithme de répartition de charge, gestion des sessions, limites de débit. Ces annotations s'appliquent par route, ce qui permet des configurations différentes par application sans toucher à la configuration globale du routeur.

Les annotations les plus utiles en production :

| Annotation | Rôle |
|---|---|
| `timeout` | Délai max avant de renvoyer une erreur 504 (défaut : 30s) |
| `balance` | Algorithme de répartition : `roundrobin`, `leastconn`, `source` |
| `rate-limit-connections` | Active la limitation du nombre de connexions simultanées |
| `disable_cookies` | Désactive la persistance de session par cookie |

### 4.1 — Appliquer et gérer les annotations

```bash
# Définir un timeout à 60 secondes (utile pour les requêtes longues)
oc annotate route webapp haproxy.router.openshift.io/timeout=60s

# leastconn envoie chaque nouvelle connexion vers le pod le moins chargé
# Préférable à roundrobin pour des requêtes de durées variables
oc annotate route webapp haproxy.router.openshift.io/balance=leastconn

# Modifier une annotation existante (--overwrite obligatoire, sinon erreur)
oc annotate route webapp haproxy.router.openshift.io/timeout=45s --overwrite

# Supprimer une annotation (le tiret final supprime la clé)
oc annotate route webapp haproxy.router.openshift.io/balance-

# Vérifier toutes les annotations actives sur la route
oc describe route webapp | grep -A 10 "Annotations"
```

## Étape 5 — Exposer PostgreSQL via NodePort

### Concept

Les Routes OpenShift ne gèrent que HTTP, HTTPS et TLS avec SNI. Pour du **TCP pur** (PostgreSQL, Redis, MQTT...), il faut un autre mécanisme.

Un **NodePort** ouvre un port fixe (entre 30000 et 32767) sur **tous les nœuds** du cluster. Tout trafic arrivant sur ce port d'un nœud est redirigé vers les pods du Service correspondant.

```
Client ──TCP:30432──► Nœud worker ──► Service ──► Pod PostgreSQL
```

Sur ROSA, les nœuds sont dans un VPC AWS privé et n'ont pas d'IP publique. Le NodePort est accessible uniquement depuis l'intérieur du VPC (bastion, autre EC2) ou depuis un pod dans le cluster — ce qui est suffisant pour valider la connectivité dans ce lab.

### 5.1 — Déployer PostgreSQL

PostgreSQL est déployé via un **StatefulSet** plutôt qu'un Deployment, car il gère un état persistant. La `readinessProbe` avec `pg_isready` garantit que le pod n'est déclaré prêt que quand PostgreSQL accepte réellement des connexions.

```bash
cat > postgres.yaml << 'EOF'
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: registry.redhat.io/rhel9/postgresql-16:latest
          env:
            - name: POSTGRESQL_USER
              value: "pguser"
            - name: POSTGRESQL_PASSWORD
              value: "pgpassword"
            - name: POSTGRESQL_DATABASE
              value: "workshop"
          ports:
            - containerPort: 5432
          readinessProbe:
            exec:
              command:
                - /bin/sh
                - -c
                - pg_isready -U pguser -d workshop
            initialDelaySeconds: 10
            periodSeconds: 5
EOF

oc apply -f postgres.yaml

# Attendre que le pod soit Ready (la readinessProbe doit passer)
oc get pods -l app=postgres -w
```

### 5.2 — Créer le Service NodePort

Le champ `nodePort: 30432` fixe le port sur les nœuds. Si omis, Kubernetes en assigne un aléatoirement dans la plage 30000-32767.

```bash
cat > svc-nodeport.yaml << 'EOF'
apiVersion: v1
kind: Service
metadata:
  name: postgres-nodeport
spec:
  type: NodePort
  selector:
    app: postgres
  ports:
    - port: 5432
      targetPort: 5432
      nodePort: 30432
EOF

oc apply -f svc-nodeport.yaml

# La colonne PORT(S) doit afficher 5432:30432/TCP
oc get svc postgres-nodeport
```

### 5.3 — Tester depuis l'intérieur du cluster

Sur ROSA, la méthode fiable est de lancer un pod client temporaire à l'intérieur du cluster. Ce pod peut joindre le Service directement par son nom DNS (`postgres-nodeport`), sans passer par les nœuds.

> **`--restart=Never` est obligatoire.** Sans ce flag, OpenShift considère le pod comme un service à redémarrer automatiquement. Quand `psql` se termine (même avec succès), Kubernetes redémarre le pod en boucle → CrashLoopBackOff.

```bash
oc run pg-client --rm -it --restart=Never \
  --image=registry.redhat.io/rhel9/postgresql-16:latest \
  --env="PGPASSWORD=pgpassword" \
  -- psql -h postgres-nodeport -U pguser -d workshop -c "\l"
```

Résultat attendu : liste des bases de données incluant `workshop`.

## Étape 6 — Exposer PostgreSQL via LoadBalancer (AWS NLB)

### Concept

Le type **LoadBalancer** est la solution de production pour exposer du TCP non-HTTP hors du cluster. Sur ROSA, créer un Service `LoadBalancer` déclenche automatiquement la création d'un **Network Load Balancer AWS** via le cloud-controller-manager. Ce NLB reçoit un hostname DNS stable fourni par AWS.

```
Client ──TCP:5432──► NLB AWS ──► Service ──► Pod PostgreSQL
```

L'annotation `aws-load-balancer-internal: "true"` crée un NLB **interne au VPC**, non accessible depuis Internet. C'est la configuration recommandée pour les bases de données en production. Retirez-la uniquement si vous avez besoin d'un accès depuis Internet.

> **Attention aux coûts :** chaque Service `LoadBalancer` crée un NLB AWS qui est facturé à l'heure. Supprimez ce service impérativement à la fin du lab.

### 6.1 — Créer le Service LoadBalancer

```bash
cat > svc-lb.yaml << 'EOF'
apiVersion: v1
kind: Service
metadata:
  name: postgres-lb
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
spec:
  type: LoadBalancer
  selector:
    app: postgres
  ports:
    - name: postgres
      port: 5432
      targetPort: 5432
EOF

oc apply -f svc-lb.yaml
```

### 6.2 — Attendre le provisionnement du NLB

Le NLB AWS prend généralement 1 à 2 minutes à être provisionné. Le champ `EXTERNAL-IP` passe de `<pending>` à un hostname AWS quand c'est prêt.

```bash
# Surveiller jusqu'à ce que EXTERNAL-IP soit renseigné (Ctrl+C pour sortir)
watch -n 10 'oc get svc postgres-lb'
```

Résultat attendu une fois prêt :
```
NAME          TYPE           CLUSTER-IP    EXTERNAL-IP                          PORT(S)
postgres-lb   LoadBalancer   10.0.10.100   abc123.elb.eu-west-1.amazonaws.com   5432:31234/TCP
```

### 6.3 — Tester la connexion

Le NLB est interne au VPC, il n'est pas accessible depuis votre poste local. On teste depuis un pod dans le cluster, exactement comme pour le NodePort.

```bash
LB_HOST=$(oc get svc postgres-lb -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')
echo "LoadBalancer hostname : $LB_HOST"

oc run pg-client --rm -it --restart=Never \
  --image=registry.redhat.io/rhel9/postgresql-16:latest \
  --env="PGPASSWORD=pgpassword" \
  -- psql -h "$LB_HOST" -U pguser -d workshop -c "SELECT version();"
```

### Tableau de décision

| Méthode | Protocole | Qui gère le TLS | Cas d'usage typique |
|---|---|---|---|
| Route HTTP | HTTP | — | Dev, apps internes sans sécurité |
| Route Edge | HTTPS | Le routeur | Applications web en production |
| Route Passthrough | HTTPS | Le pod | Apps avec TLS applicatif (gRPC, mTLS) |
| Route Re-encrypt | HTTPS | Routeur + Pod | Double chiffrement requis |
| NodePort | TCP/UDP | — | Tests internes, accès depuis le VPC |
| LoadBalancer | TCP/UDP | — | Services TCP en production (NLB AWS) |

### Schéma de décision

```
Mon application parle HTTP/HTTPS ?
├── OUI → Route
│         ├── Pas de TLS nécessaire    → Route HTTP
│         ├── TLS géré par OpenShift   → Route Edge      (le plus courant)
│         ├── TLS géré par le pod      → Route Passthrough
│         └── TLS des deux côtés       → Route Re-encrypt
│
└── NON (TCP pur : PostgreSQL, Redis, MQTT...)
          ├── Test / accès depuis VPC  → NodePort
          └── Production               → LoadBalancer (NLB AWS)
```

## Pièges fréquents

- **Route sans TLS en production** → les données circulent en clair sur le réseau
- **`insecureEdgeTerminationPolicy` non défini** → HTTP et HTTPS coexistent ; préférez `Redirect`
- **Passthrough sans TLS dans le pod** → le routeur ne peut pas traiter le trafic et renvoie une erreur SSL
- **`targetPort` incorrect dans la Route** → la route est créée mais renvoie 503 ; le port doit correspondre au `containerPort` du pod
- **`--overwrite` manquant** → `oc annotate` échoue si l'annotation existe déjà
- **`oc run` sans `--restart=Never`** → CrashLoopBackOff dès que la commande se termine, même avec succès
- **`ubi-minimal` pour tester curl** → `curl` est absent de cette image ; utiliser `ubi9/ubi:latest`
- **`nginx:alpine` sur OpenShift** → Permission denied sur `/var/cache/nginx/` ; utiliser `nginxinc/nginx-unprivileged:alpine`
- **`ExternalIP` vide sur ROSA** → les nœuds n'ont que des `InternalIP` ; tester via `oc run` depuis le cluster
- **Oublier de supprimer le LoadBalancer** → le NLB AWS continue d'être facturé après le lab

## Annexe — Hints de dépannage

<details>
<summary>💡 Route bloquée / erreur 503 ?</summary>

Une erreur 503 signifie que la route existe mais que le routeur ne trouve pas de pod disponible. Les causes les plus fréquentes : pods non Running, labels incorrects entre le pod et le Service, ou mauvais port.

```bash
# Les pods sont-ils Running ?
oc get pods -l app=webapp

# Le Service a-t-il des endpoints ? (vide = labels incorrects)
oc get endpoints webapp

# Le targetPort de la route correspond-il au port du Service ?
oc get route webapp -o jsonpath='{.spec.port.targetPort}' && echo
oc get svc webapp -o jsonpath='{.spec.ports[0].port}' && echo

# Tester directement depuis l'intérieur du cluster
oc run test --rm -it --restart=Never \
  --image=registry.access.redhat.com/ubi9/ubi:latest \
  -- curl http://webapp:8080
```

</details>

<details>
<summary>💡 LoadBalancer reste en Pending ?</summary>

Si le champ `EXTERNAL-IP` reste `<pending>` plus de 3 minutes, le cloud-controller-manager n'a pas pu créer le NLB. Les causes les plus fréquentes : droits IAM insuffisants, quota de NLB AWS atteint, ou problème de configuration de sous-réseau.

```bash
# Voir les événements récents sur le Service
oc describe svc postgres-lb | tail -20

# Voir les logs du composant qui crée le NLB
CCM_POD=$(oc get pods -n kube-system -l app=cloud-controller-manager \
  -o jsonpath='{.items[0].metadata.name}')
oc logs -n kube-system "$CCM_POD" --tail=50 | grep -i "loadbalancer"
```

</details>

<details>
<summary>💡 Tester un port TCP sans nc installé ?</summary>

`nc` (netcat) n'est pas toujours disponible. Deux alternatives sans dépendance externe :

```bash
# Méthode 1 — bash natif, aucune installation requise
timeout 5 bash -c 'echo > /dev/tcp/<host>/5432' 2>/dev/null \
  && echo "Port ouvert" || echo "Port fermé"

# Méthode 2 — python3, affiche le message d'erreur exact en cas d'échec
python3 -c "
import socket, sys
s = socket.socket(); s.settimeout(5)
try:
    s.connect(('<host>', 5432)); print('Port ouvert'); s.close()
except Exception as e:
    print(f'Échec : {e}'); sys.exit(1)
"
```

</details>

ROUTER_POD=$(oc get pods -n openshift-ingress -o name | head -1)

# Logs généraux du routeur
