# Lab 08 - Exposition avancée des applications et sécurité des workloads

---
# Contexte

La plateforme **DocuShare** extrait une fonctionnalité du monolithe vers un nouveau microservice interne : `document-metadata-api`

Ce service fournit les métadonnées des documents aux autres composants internes :

* portail utilisateur ;
* moteur de recherche ;
* service d’audit ;
* workflow d’archivage.

Ce service ne doit pas être exposé publiquement. Il doit seulement être accessible aux autres workloads du namespace.

L’équipe sécurité impose également que le workload respecte les standards OpenShift :

* pas d’exécution en root ;
* pas d’élévation de privilèges ;
* suppression des capabilities Linux inutiles ;
* compatibilité avec les UID ranges OpenShift.

---

# Objectif

Déployer un microservice interne puis vérifier qu’il est :

* accessible via un `Service` interne ;
* exécuté avec un UID non-root ;
* conforme au modèle de sécurité OpenShift ;
* toujours fonctionnel après durcissement.

---

# Étape 1 - Déployer le microservice interne

## Mission

Créer un `Deployment` nommé :

```text
document-metadata-api
```

Le workload doit :

- utiliser l’image compatible OpenShift :
  `registry.redhat.io/ubi9/nodejs-20`
- écouter sur le port `8080` ;
- répondre avec le texte `metadata api ok` ;
- avoir le label `app: metadata-api`.
---

<details>
<summary>💡 Hint - Deployment</summary>

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: document-metadata-api
  namespace: docushare-dev
spec:
  replicas: 1
  selector:
    matchLabels:
      app: metadata-api
  template:
    metadata:
      labels:
        app: metadata-api
    spec:
      containers:
        - name: api
          image: registry.redhat.io/ubi9/nodejs-20
          command:
            - /bin/bash
            - -c
          args:
            - |
              echo "const http=require('http');
              http.createServer((req,res)=>res.end('metadata api ok')).listen(8080,'0.0.0.0');" > app.js;
              node app.js
          ports:
            - containerPort: 8080
```

</details>

# Étape 5 - Appliquer un SecurityContext

## Mission

Modifier le `Deployment` `document-metadata-api` pour appliquer les règles de sécurité suivantes :

* le pod doit s’exécuter en non-root (OpenShift lance déjà en non-root, mais nous l’imposons aussi applicativement via SecurityContext.)
* l’élévation de privilèges doit être interdite ;
* toutes les capabilities Linux doivent être supprimées.


---

<details>
<summary>💡 Hint - Deployment sécurisé</summary>

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: document-metadata-api
  namespace: docushare-dev
spec:
  replicas: 1
  selector:
    matchLabels:
      app: metadata-api
  template:
    metadata:
      labels:
        app: metadata-api
    spec:
      securityContext:
        runAsNonRoot: true
      containers:
        - name: api
          image: registry.redhat.io/ubi9/nodejs-20
          command:
            - /bin/bash
            - -c
          args:
            - |
              echo "const http=require('http');
              http.createServer((req,res)=>res.end('metadata api ok')).listen(8080,'0.0.0.0');" > app.js;
              node app.js
          ports:
            - containerPort: 8080
          securityContext:
            allowPrivilegeEscalation: false
            capabilities:
              drop:
                - ALL
```

</details>

---

## Rappel sécurité du SecurityContext

```yaml
securityContext:
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
```

### Pourquoi l’ajouter ?

#### `allowPrivilegeEscalation: false`

Empêche un processus du conteneur d’obtenir plus de privilèges après démarrage.

But :

* limiter les escalades de privilèges ;
* réduire l’impact d’une faille applicative ;
* respecter les standards sécurité plateforme.

#### `capabilities: drop: ALL`

Supprime les privilèges Linux additionnels non nécessaires.

But :

* réduire la surface d’attaque ;
* empêcher certaines actions système sensibles ;
* appliquer le principe du moindre privilège.

### À retenir

```text
runAsNonRoot = ne pas lancer en root
allowPrivilegeEscalation = pas de montée de privilèges
drop ALL = retirer les privilèges inutiles
```

Un workload moderne doit fonctionner même avec ces restrictions.

Le lab est réussi si :

# Ce qu’il faut retenir

* un service interne doit être publié proprement via un `Service` ;
* OpenShift impose des UID dynamiques pour limiter les risques ;
* un workload compatible OpenShift ne doit pas dépendre de root ;
* `SecurityContext` permet de durcir l’exécution des conteneurs ;
* un workload sécurisé doit rester fonctionnel après durcissement.
