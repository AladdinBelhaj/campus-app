# Lab — Expose HTTP and non-HTTP Applications to External Access
---

## Contexte

Votre équipe développe une plateforme composée de :

- Une **application web** (HTTP/HTTPS) — à exposer via Route
- Une **base PostgreSQL** (TCP non-HTTP) — à exposer via NodePort et LoadBalancer
- Une **application avec TLS natif** — à exposer via Route Passthrough

---

## Objectifs

À travers ce lab, vous allez manipuler :

- Créer et configurer des **Routes HTTP et HTTPS**
- Comprendre les modes **Edge / Passthrough / Re-encrypt**
- Exposer des services **non-HTTP** via NodePort et LoadBalancer
- Utiliser les **annotations HAProxy** pour contrôler le trafic
- Vérifier l'accès externe depuis `curl` et le navigateur

---

## Étape 1 — Exposer une application HTTP simple

### Mission

Déployer une application web et l'exposer en HTTP via une Route OpenShift.

### 1.1 — Déployer l'application et son Service

```yaml
# webapp.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
        - name: webapp
          image: registry.access.redhat.com/ubi9/python-311:latest
          command:
            - /bin/sh
            - -c
            - |
              python3 -m http.server 8080
          ports:
            - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: webapp
spec:
  selector:
    app: webapp
  ports:
    - port: 8080
      targetPort: 8080
```

```bash
oc apply -f webapp.yaml
oc get pods -l app=webapp
oc get svc webapp
```

### 1.2 — Créer la Route HTTP

```bash
# Méthode rapide — expose le service directement
oc expose service webapp

# Vérifier la route créée
oc get route webapp
oc get route webapp -o jsonpath='{.spec.host}'
```

### 1.3 — Tester l'accès externe

```bash
# Récupérer l'URL complète
ROUTE=$(oc get route webapp -o jsonpath='{.spec.host}')
echo "URL : http://$ROUTE"

# Tester avec curl
curl -v http://$ROUTE

# Vérifier le code HTTP retourné
curl -o /dev/null -s -w "%{http_code}\n" http://$ROUTE
```

### Validation attendue

- Route créée avec un hostname automatique
- `curl` retourne HTTP `200`
- La page s'affiche dans le navigateur

## Étape 2 — Route HTTPS — Terminaison Edge

### Principe

```
Client ──HTTPS──► Routeur OpenShift ──HTTP──► Pod
```

Le trafic est chiffré jusqu'au routeur. Le pod reçoit le trafic en clair.
Le certificat TLS est géré par OpenShift, pas par le pod.

### 2.1 — Générer un certificat auto-signé

```bash
# Créer la clé privée et le certificat
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout tls.key \
  -out tls.crt \
  -subj "/CN=webapp-edge.apps.rosa-cluster.example.com/O=Workshop"

# Vérifier le certificat
openssl x509 -in tls.crt -text -noout | grep -E "Subject|Not"

# Créer le Secret TLS dans OpenShift
oc create secret tls webapp-tls \
  --cert=tls.crt \
  --key=tls.key

oc get secret webapp-tls
```

### 2.2 — Créer la Route Edge

```bash
# Méthode CLI — la plus simple
oc create route edge webapp-edge \
  --service=webapp \
  --cert=tls.crt \
  --key=tls.key \
  --insecure-policy=Redirect

oc get route webapp-edge
```

```yaml
# Équivalent YAML — route-edge.yaml
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: webapp-edge
spec:
  to:
    kind: Service
    name: webapp
  port:
    targetPort: 8080
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect
```

### 2.3 — Tester l'accès HTTPS

```bash
ROUTE=$(oc get route webapp-edge -o jsonpath='{.spec.host}')

# HTTPS avec certificat auto-signé (ignorer la vérification)
curl -k https://$ROUTE

# Vérifier la redirection HTTP → HTTPS
curl -v http://$ROUTE
# Attendu : 301 Moved Permanently vers https://

# Voir le certificat présenté
echo | openssl s_client -connect $ROUTE:443 2>/dev/null \
  | openssl x509 -noout -subject
```

> **Note :** Le navigateur affichera un avertissement de sécurité
> car le certificat est auto-signé. C'est normal en environnement de test.
> En production, utilisez un certificat signé par une CA reconnue.

### Validation attendue

- HTTPS fonctionne avec `curl -k`
- HTTP redirige automatiquement vers HTTPS (`301`)
- Le certificat est visible dans le navigateur

## Étape 3 — Route HTTPS — Terminaison Passthrough

### Principe

```
Client ──HTTPS──► Routeur OpenShift ──HTTPS──► Pod
```

Le routeur ne déchiffre pas le trafic. Le pod gère lui-même le certificat TLS.
Utile pour les applications qui imposent leur propre TLS (bases de données, gRPC...).

### 3.1 — Déployer une application avec TLS natif

```yaml
# nginx-tls.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    events {}
    http {
      server {
        listen 8443 ssl;
        ssl_certificate     /etc/nginx/ssl/tls.crt;
        ssl_certificate_key /etc/nginx/ssl/tls.key;
        location / {
          default_type text/plain;
          return 200 "Hello from NGINX with native TLS!";
        }
        location /health {
          default_type text/plain;
          return 200 "OK";
        }
      }
    }
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-tls
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx-tls
  template:
    metadata:
      labels:
        app: nginx-tls
    spec:
      containers:
        - name: nginx
          image: nginx:alpine
          ports:
            - containerPort: 8443
          volumeMounts:
            - name: tls
              mountPath: /etc/nginx/ssl
              readOnly: true
            - name: config
              mountPath: /etc/nginx/nginx.conf
              subPath: nginx.conf
      volumes:
        - name: tls
          secret:
            secretName: webapp-tls
        - name: config
          configMap:
            name: nginx-config
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-tls
spec:
  selector:
    app: nginx-tls
  ports:
    - port: 8443
      targetPort: 8443
```

```bash
oc apply -f nginx-tls.yaml

# Attendre que le pod soit Running
oc get pods -l app=nginx-tls -w

# Tester en interne depuis un pod temporaire
oc run test-curl --rm -it \
  --image=registry.access.redhat.com/ubi9/ubi-minimal:latest \
  -- curl -k https://nginx-tls:8443
```

### 3.2 — Créer la Route Passthrough

```bash
oc create route passthrough nginx-passthrough \
  --service=nginx-tls \
  --port=8443

oc get route nginx-passthrough
```

### 3.3 — Tester l'accès externe

```bash
ROUTE=$(oc get route nginx-passthrough -o jsonpath='{.spec.host}')

# Le pod gère le TLS directement
curl -k https://$ROUTE
# Attendu : "Hello from NGINX with native TLS!"

# Voir que le certificat vient du pod (pas du routeur)
echo | openssl s_client -connect $ROUTE:443 2>/dev/null \
  | openssl x509 -noout -subject -issuer
```

### Validation attendue

- Le pod gère lui-même le TLS
- OpenShift ne déchiffre pas le trafic
- `curl -k` retourne la réponse du pod

## Étape 4 — Annotations de Route HAProxy

### Mission

Contrôler finement le comportement du routeur via des annotations.

### 4.1 — Appliquer des annotations

```bash
# Timeout personnalisé (défaut : 30s)
oc annotate route webapp \
  haproxy.router.openshift.io/timeout=60s

# Algorithme de load balancing
# Valeurs valides : roundrobin, leastconn, source, random
oc annotate route webapp \
  haproxy.router.openshift.io/balance=leastconn

# Taille maximale du corps de requête
oc annotate route webapp \
  haproxy.router.openshift.io/request-limit=10m

# Activer le rate limiting
oc annotate route webapp \
  haproxy.router.openshift.io/rate-limit-connections=true

# Gestion des cookies de session
oc annotate route webapp \
  haproxy.router.openshift.io/disable_cookies=false
```

### 4.2 — Vérifier et modifier les annotations

```bash
# Via describe
oc describe route webapp | grep -A 15 "Annotations"

# Via yaml
oc get route webapp -o yaml | grep -A 10 annotations

# Modifier une annotation existante (flag --overwrite obligatoire)
oc annotate route webapp \
  haproxy.router.openshift.io/timeout=45s \
  --overwrite

# Supprimer une annotation (ajouter - à la fin du nom)
oc annotate route webapp \
  haproxy.router.openshift.io/balance-
```

### Validation attendue

- Annotations visibles dans `oc describe route webapp`
- Aucune erreur lors de l'application ou de la suppression

## Étape 5 — Exposer PostgreSQL (non-HTTP) via NodePort

### Pourquoi NodePort ?

Les Routes OpenShift ne supportent que HTTP/HTTPS et SNI (TCP avec TLS).
Pour du **TCP pur sans TLS**, il faut utiliser NodePort ou LoadBalancer.

### 5.1 — Déployer PostgreSQL

```yaml
# postgres.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: registry.redhat.io/rhel9/postgresql-16:latest
          env:
            - name: POSTGRESQL_USER
              value: "pguser"
            - name: POSTGRESQL_PASSWORD
              value: "pgpassword"
            - name: POSTGRESQL_DATABASE
              value: "workshop"
          ports:
            - containerPort: 5432
          readinessProbe:
            exec:
              command:
                - /bin/sh
                - -c
                - pg_isready -U pguser -d workshop
            initialDelaySeconds: 10
            periodSeconds: 5
```

```bash
oc apply -f postgres.yaml

# Attendre que le pod soit Ready
oc get pods -l app=postgres -w

# Vérifier les logs
oc logs -l app=postgres --tail=20
```

### 5.2 — Créer un Service NodePort

```yaml
# svc-nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres-nodeport
spec:
  type: NodePort
  selector:
    app: postgres
  ports:
    - port: 5432
      targetPort: 5432
      nodePort: 30432
```

```bash
oc apply -f svc-nodeport.yaml

# Vérifier le service — doit afficher 5432:30432/TCP
oc get svc postgres-nodeport
```

### 5.3 — Tester la connexion

```bash
# Récupérer l'IP externe d'un nœud worker
NODE_IP=$(oc get nodes -l node-role.kubernetes.io/worker \
  -o jsonpath='{.items[0].status.addresses[?(@.type=="ExternalIP")].address}')
echo "IP nœud worker : $NODE_IP"

# Test TCP simple
nc -zv $NODE_IP 30432

# Tester depuis un pod temporaire dans le cluster (méthode fiable)
oc run pg-client --rm -it \
  --image=registry.redhat.io/rhel9/postgresql-16:latest \
  --env="PGPASSWORD=pgpassword" \
  -- psql -h postgres-nodeport -U pguser -d workshop -c "\l"
```

> **Note :** L'accès direct via l'IP du nœud peut être bloqué par les
> Security Groups AWS. La méthode `oc run` (test interne) est toujours fiable.

### Validation attendue

- Service de type `NodePort` créé avec port `30432`
- Connexion TCP établie (nc ou oc run)
- Requête PostgreSQL exécutée avec succès

## Étape 6 — Exposer PostgreSQL via LoadBalancer (AWS NLB)

### Pourquoi LoadBalancer ?

Sur ROSA (AWS), un Service `LoadBalancer` crée automatiquement un
**Network Load Balancer** AWS avec un hostname stable.
Plus robuste que NodePort pour la production.

### 6.1 — Créer le Service LoadBalancer

```yaml
# svc-lb.yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres-lb
  annotations:
    # AWS NLB au lieu d'ELB classique
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    # Restreindre à l'accès interne VPC (recommandé en production)
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
spec:
  type: LoadBalancer
  selector:
    app: postgres
  ports:
    - name: postgres
      port: 5432
      targetPort: 5432
```

```bash
oc apply -f svc-lb.yaml

# Surveiller le provisionnement du LoadBalancer (1 à 2 minutes)
watch oc get svc postgres-lb
```

### 6.2 — Surveiller le provisionnement

```bash
# Voir les events liés au LoadBalancer
oc describe svc postgres-lb | grep -A 10 Events

# Vérifier l'assignation de l'EXTERNAL-IP
oc get svc postgres-lb -o jsonpath='{.status.loadBalancer.ingress}'
```

### Résultat attendu

```
NAME          TYPE           CLUSTER-IP    EXTERNAL-IP                         PORT(S)
postgres-lb   LoadBalancer   10.0.10.100   abc123.elb.eu-west-1.amazonaws.com  5432:31234/TCP
```

### 6.3 — Tester la connexion externe

```bash
# Récupérer le hostname du LoadBalancer
LB_HOST=$(oc get svc postgres-lb \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')
echo "LoadBalancer : $LB_HOST"

# Tester la connexion TCP
nc -zv $LB_HOST 5432

# Connexion PostgreSQL
PGPASSWORD=pgpassword psql \
  -h $LB_HOST \
  -U pguser \
  -d workshop \
  -c "SELECT version();"
```

> **Note :** Avec `internal=true`, le LoadBalancer est accessible uniquement
> depuis le VPC AWS. Pour un accès externe, retirez l'annotation `internal`.

### Validation attendue

- Hostname AWS assigné dans `EXTERNAL-IP`
- Connexion TCP établie via `nc`
- NLB visible dans la console AWS (EC2 → Load Balancers)

## Étape 7 — Récapitulatif et comparaison

### 7.1 — Voir toutes les ressources créées

```bash
# Toutes les routes
oc get routes -o wide

# Tous les services avec leur type
oc get svc

# Résumé complet
oc get routes,svc -o wide
```

### 7.2 — Tableau comparatif

| Méthode | Protocole | Terminaison TLS | Cas d'usage | Coût AWS |
|---|---|---|---|---|
| Route HTTP | HTTP | Aucune | Dev, apps internes | Gratuit |
| Route Edge | HTTPS | Au routeur | Apps web production | Gratuit |
| Route Passthrough | HTTPS/TLS | Au pod | Apps gérant leur TLS | Gratuit |
| Route Re-encrypt | HTTPS | Routeur + Pod | Double chiffrement | Gratuit |
| NodePort | TCP/UDP | Aucune | Dev/test, accès nœud direct | Gratuit |
| LoadBalancer | TCP/UDP | Aucune | Production non-HTTP | NLB payant |

### 7.3 — Schéma de décision

```
Mon application est-elle HTTP/HTTPS ?
├── OUI → Utiliser une Route
│         ├── Pas de TLS nécessaire    → Route HTTP
│         ├── TLS géré par OpenShift   → Route Edge
│         ├── TLS géré par le pod      → Route Passthrough
│         └── TLS des deux côtés       → Route Re-encrypt
│
└── NON (TCP pur, UDP, PostgreSQL...)
          ├── Dev / Test               → NodePort
          └── Production               → LoadBalancer (NLB AWS)
```

## Points clés à retenir

| Concept | Détail |
|---|---|
| Route HTTP | `oc expose svc` — le plus simple, pas de TLS |
| Route Edge | TLS terminé au routeur, pod reçoit HTTP |
| Route Passthrough | TLS de bout en bout, pod gère le certificat |
| Route Re-encrypt | TLS routeur → pod avec re-chiffrement différent |
| `insecureEdgeTerminationPolicy` | `Allow`, `Redirect`, `None` |
| NodePort | TCP direct sur IP nœud, ports 30000-32767 |
| LoadBalancer | NLB AWS créé automatiquement sur ROSA |
| Annotations HAProxy | Contrôle timeout, balance, rate-limit, cookies |

---

## Pièges fréquents

- **Route sans TLS en production** → données en clair sur le réseau
- **NodePort exposé publiquement** → vérifier les Security Groups AWS
- **LoadBalancer sans annotation `internal`** → NLB public exposé à Internet
- **Passthrough sans TLS dans le pod** → erreur SSL au niveau du routeur
- **Port `targetPort` incorrect** → route créée mais connexion refusée (503)
- **Oublier de supprimer le LoadBalancer** → coût AWS continu après le lab
- **`--overwrite` manquant** → erreur lors de la modification d'une annotation

<details>
<summary>💡 Hint — Différence Edge / Passthrough / Re-encrypt</summary>

```
Edge        : Client ──HTTPS──► Routeur ──HTTP──► Pod
              Le routeur déchiffre et retransmet en clair

Passthrough : Client ──HTTPS──► Routeur ──HTTPS──► Pod
              Le routeur transmet sans déchiffrer (SNI routing)

Re-encrypt  : Client ──HTTPS──► Routeur ──HTTPS──► Pod
              Le routeur déchiffre et re-chiffre avec un autre certificat
```
