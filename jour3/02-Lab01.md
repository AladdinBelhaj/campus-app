# Étape 2 - Déployer la base PostgreSQL de DocuShare

## Mission

L’application DocuShare doit disposer d’une base PostgreSQL persistante pour stocker :

* les utilisateurs ;
* l’index documentaire ;
* les métadonnées ;
* l’historique.

Déployez cette base dans OpenShift.

---

## Contraintes attendues

Vous devez créer :

### Stockage

* un `PersistentVolumeClaim`
* nom : `pgdata`
* taille : `2Gi`
* mode d’accès : `ReadWriteOnce`
* `StorageClass` : `ocs-storagecluster-ceph-rbd`

### Base de données

* un workload de type `StatefulSet`
* image : `registry.redhat.io/rhel9/postgresql-16`

* variables minimales :

```text
POSTGRESQL_USER
POSTGRESQL_PASSWORD
POSTGRESQL_DATABASE
```

### Exposition interne

* un `Service`
* port `5432`

---

## Validation attendue

Montrez :

* le PVC `pgdata` en `Bound`
* le pod PostgreSQL en `Running`
* les logs PostgreSQL cohérents
* le service créé

# Étape 4 - Déployer deux instances applicatives DocuShare

## Mission

Vous allez maintenant simuler deux instances frontend de l’application **DocuShare**.

Ces deux instances représentent deux pods applicatifs distincts capables d’accéder aux mêmes documents utilisateurs.

Pour garantir ce comportement, les deux instances devront monter le même volume partagé `documents`.

Vous devez créer deux **Deployments** :

- `docushare-a`
- `docushare-b`

---

## Pourquoi un Deployment ici ?

Dans ce cas d’usage, les instances frontend sont **stateless** :

- elles ne stockent pas leurs données localement ;
- elles utilisent le volume partagé pour les documents ;
- OpenShift peut les recréer automatiquement en cas de panne ;
- elles peuvent être scalées facilement.

Le `Deployment` est donc le bon choix.

---

## Contraintes attendues

Créer deux `Deployment` distincts utilisant l’image :

```text
registry.access.redhat.com/ubi9/ubi-minimal:latest
````

Chaque deployment doit contenir :

### Réplicas

* `1 replica`

### Labels

Chaque application doit avoir son propre label :

* `app: docushare-a`
* `app: docushare-b`

### Conteneur

Le conteneur doit :

* rester actif ;
* utiliser une commande simple ;
* monter le volume partagé.

### Stockage

Le PVC suivant doit être monté :

```text
documents
```

Chemin de montage recommandé :

```text
/documents
```

---

## Ce que vous devez construire

Créer :

* un deployment `docushare-a`
* un deployment `docushare-b`

Les deux doivent partager :

* le même PVC
* le même mountPath

---

## Validation attendue

Montrez :

* les deux deployments créés ;
* deux pods en `Running` ;
* le PVC `documents` utilisé par les deux ;
* le montage visible dans les pods.

---

## Vérifications utiles

```bash
oc get deploy
oc get pods
oc describe pod <pod-name>
```

# Étape 7 - Créer un snapshot avant mise à jour

## Mission

Avant une future mise à jour de PostgreSQL, l’équipe exploitation demande une sauvegarde rapide.


* Créez un `VolumeSnapshot`  nommé `pgdata-snapshot` du PVC `pgdata`, en utilisant la VolumeSnapshotClass `ocs-storagecluster-rbdplugin-snapclass`.

---

## Validation attendue

Montrez :

* la ressource `VolumeSnapshot`
* un état prêt / usable
* la référence au PVC `pgdata`

---

<details>
<summary>💡 Hint - Squelette VolumeSnapshot</summary>

```yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: ...
spec:
  volumeSnapshotClassName: ...
  source:
    persistentVolumeClaimName: pgdata
```

</details>
